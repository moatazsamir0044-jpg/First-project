A fully functional prototype of a modernised BirdNest website has been built and is ready for review. The goal of this meeting is to agree on a safe, controlled way to bring these improvements into the live site — on a timeline the tech team owns entirely.

First What This Is NOT Asking For

Concern The Reality
"Move the live database to a new system" No migration needed. All features are additive — new columns at most, never replacing existing data.
"Switch the payment gateway" No change to the gateway. The existing provider stays exactly as-is. No client payment touches the prototype until tech signs off.
"Give full codebase access" Not required. Two paths work: (A) features as pull requests the tech team reviews line-by-line, or (B) full spec + code handed over for the team to implement themselves.

Plan Proposed Phased Rollout

Phase 1Week 1–2

Review

Owner: Tech Director
  • Spin up a private staging environment using the prototype code
  • Stripe runs in test mode — zero real money, zero client data
  • Tech director reviews all code before anything proceeds
  • No connection to the live database; uses a separate test database
The tech director controls everything. He can reject, modify, or pause at any step.
Phase 2Week 2–3

Select & Approve Features

Owner: Tech Director + Ecommerce Manager
  • Walk through the feature list together
  • Choose which features to adopt first (suggested: checkout flow, search filters, security headers)
  • Tech director decides: implement from spec, or review submitted PRs
Phase 3Week 3–6

Staged Rollout

Owner: Tech Director
  • Features merged one at a time, each fully tested before the next
  • Staging environment verified by ecommerce manager for business outcomes
  • No feature goes live without tech director approval
Phase 4Go-Live

Live

When tech director confirms ready
  • Real payment keys enabled only after independent security confirmation
  • DNS, domain, and hosting remain under tech team control throughout

Value What the Tech Director Gets

  • Full control over every merge Nothing reaches production without his review. He is the gatekeeper at every step.
  • No big-bang risk One feature at a time. Roll back any single feature without affecting others.
  • Security-hardened codebase CSP headers, HSTS, RLS on all DB tables, scrypt password hashing — all built in and documented.
  • Time saved Features are already built and tested. He reviews and integrates rather than building from scratch.

The only deliverable needed from this meeting:
Agree on which 2–3 features to adopt first, name who owns the staging setup, and set a date for the Phase 1 review.

Everything else follows from that single agreement.

Appendix Security Architecture Summary

Content Security Policy (CSP) Nonce-based. Explicitly allowlists only Stripe, Google, Mapbox, and Vercel. Blocks all other script injection automatically.
HSTS with Preload Forces HTTPS on all connections for 2 years. Submitted for browser preload lists. Required for PCI compliance.
Clickjacking Protection X-Frame-Options: DENY on every response. Prevents embedding the site in malicious iframes.
Row-Level Security (RLS) Enabled on all 8 database tables. No data is accessible through the public API without an explicit policy.
Secure Password Hashing scrypt algorithm (NIST-recommended), constant-time comparison. Stolen hashes cannot be reversed.
Rate Limiting + Enumeration Safety Auth routes are rate-limited per IP. Registration never reveals whether an email already exists.
An independent security review is recommended before enabling real payments — this is built into Phase 4. No card data is ever stored; Stripe handles all card tokenisation.
