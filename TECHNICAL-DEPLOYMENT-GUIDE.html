1 · System Overview

BirdNest is a server-rendered Next.js 14 (App Router) application for holiday rentals. Front-end pages and the backend API live in one Next.js project, deployed together to Vercel. Data is in PostgreSQL on Supabase, accessed via Prisma. Payments via Stripe, email via Resend, an optional chatbot via Anthropic.

6 · Authentication (lib/auth.ts)

  • NextAuth v5. Credentials (email/password) + Google OAuth. Session strategy = JWT; role added to session.
  • trustHost: true; secret = NEXTAUTH_SECRET ?? AUTH_SECRET.
  • Passwords hashed with Node's built-in scrypt (no native deps — serverless-friendly), constant-time verify.
  • Register endpoint: 8+ chars upper/lower/number, rate-limited, enumeration-safe errors.

7 · Payments — Current vs Target

Backend payment path is fully built. The checkout UI is NOT wired to it — it simulates payment.

Current in components/booking/QuickCheckout.tsx — card fields are plain inputs, not Stripe Elements:

// Simulated until live keys are wired.
await new Promise(r => setTimeout(r, 1500))   // ◀ FAKE: no charge, no booking
onComplete(form)

Target flow to implement:

  1. POST /api/payments/create-intent → receive clientSecret.
  2. Mount Stripe Elements (@stripe/react-stripe-js) with the clientSecret; replace plain card inputs with <PaymentElement>.
  3. stripe.confirmPayment(...).
  4. POST /api/bookings with stripePaymentId (status starts pending).
  5. Stripe fires payment_intent.succeeded → webhook flips booking to confirmed + sends email.

Steps 1, 4, 5 already exist server-side. The remaining work is steps 2–3 plus two fetch calls.

8 · Security (already in place)

  • middleware.ts: per-request nonce CSP + HSTS (2y, preload), X-Frame-Options: DENY, nosniff, Referrer-Policy, locked Permissions-Policy. CSP allowlists Stripe/Mapbox/Google/Unsplash/Vercel.
  • poweredByHeader: false; all inputs Zod-validated; rate limits on register/booking/payment.
  • Stripe webhook verifies signature against STRIPE_WEBHOOK_SECRET.
  • Supabase RLS enabled on all tables; Prisma connects as owner (bypasses RLS by design — anon API locked).

P0 Checkout simulates payment

P0 Vercel env vars not set

See §10 — nothing runtime works without these.

P1 Prisma migration drift

  • Booking.nationality is in the schema + live DB but no migration creates it — add a migration so a fresh migrate deploy is correct.
  • Live DB was built from the schema directly; its _prisma_migrations doesn't list the 3 repo migrations. Baseline it: prisma migrate resolve --applied <name> for each.

P1 In-memory rate limiter

lib/rate-limit.ts uses a process-local Map — ineffective across serverless instances. Move to Upstash/Redis.

P2 Pre-launch & housekeeping

  • Independent security review / pen-test; connect domain + DNS; verify Resend deliverability; native review of Arabic strings.
  • Duplicate lockfiles — keep one. vercel.json region iad1 (US) vs DB in EU — consider moving Vercel to an EU region (e.g. fra1) for latency.

11 · Deployment Runbook (to first live booking)

  1. Wire pages to the DB (P0 #1) — verify seeded listings render on /listings.
  2. Wire Stripe into checkout (P0 #2).
  3. Add nationality migration; migrate resolve --applied the 3 existing migrations against the live DB.
  4. Set env vars in Vercel (§10) for Production.
  5. Add the Stripe webhook endpoint → copy STRIPE_WEBHOOK_SECRET into Vercel.
  6. Redeploy on Vercel.
  7. E2E test on the prod URL: browse → listings appear; book with a Stripe test card → intent created → booking row written → webhook confirms → email received → shows in /account.
  8. Switch Stripe to live keys, connect domain + DNS, re-test once.
  9. Schedule the security review before announcing.

12 · What the Dev Team Should Start Doing Now

This week (unblocks go-live):

  1. Assign one owner to DB-wiring (P0 #1) and one to Stripe checkout (P0 #2) — independent, run in parallel.
  2. Vercel owner: set env vars (§10) + add the Stripe webhook (≈1 hour; blocks all runtime testing).
  3. Add the nationality migration and baseline the live DB.

Immediately, as standing practice:

  1. Resolve the lockfile / package-manager split.
  2. Stand up a Vercel Preview/staging env with its own Supabase project/branch — never test against prod data.
  3. Move the rate limiter to Upstash.
  4. Add tests around the booking + payment path and wire CI on PRs.
  5. Keep the branch → PR → squash-merge workflow off main.

Before public launch:

  1. Independent security review / pen-test.
  2. Verify Resend deliverability (SPF/DKIM on sending domain).
  3. Native-speaker review of Arabic content.
  4. Build a minimal admin UI on the existing /api/admin/* routes for non-developers.
Bottom line
The engine is built: data model, full API, auth, security, and a seeded production database all exist and work. What's left is wiring the front-end to the database, connecting the checkout to the existing Stripe backend, and setting the Vercel environment variables — plus a migration baseline. These are well-scoped integration tasks, not new systems. With focused effort the site can take its first real, paid booking within roughly one week.
